# 产品需求文档（PRD）— 学生"一站式"自主管理过程管理系统

> 文档版本：v1.0 | 创建日期：2026-06-05 | 状态：评审中
> 产品负责人：团委/学工处 | 技术负责人：待定

## 一、项目背景

### 1.1 问题陈述

高校学生事务管理长期依赖纸质表单、Excel台账和微信沟通，导致入团申请、社团活动、社区自治、勤工助学四大核心业务流程碎片化严重——数据孤岛、考勤不精准、统计滞后、档案缺失、协作低效五大痛点交织，学生需要"跑多个办公室、填多份表、等很久"才能完成一项事务，管理部门则陷入"汇总靠手工、审批靠记忆、归档靠文件夹"的低效循环。

### 1.2 现状分析

| 痛点 | 具体表现 | 影响范围 |
|------|----------|----------|
| 流程碎片化 | 入团申请走纸质、审批走微信、档案存Excel，数据孤岛严重 | 团委、学生工作处 |
| 考勤不精准 | 社团活动签到靠手写、勤工助学考勤靠打卡机，易代签漏签 | 社团指导中心、勤工助学办 |
| 统计滞后 | 月报/季报靠人工汇总，数据延迟1-2周 | 各管理部门 |
| 档案缺失 | 学生毕业时缺乏完整过程性记录，评优推荐缺乏数据支撑 | 学生个人、辅导员 |
| 协作低效 | 多部门业务交叉，信息传递依赖线下会议和群消息 | 全校师生 |

### 1.3 项目发起动因

1. **政策驱动**：教育部推进高校治理数字化转型，要求学生管理过程可追溯、数据可分析
2. **需求迫切**：团委每学期处理200+入团申请、学工处管理500+勤工助学岗位，纯人工管理已不可持续
3. **技术成熟**：Vue3 + Go + SQLite技术栈可低成本快速交付MVP，验证业务价值
4. **用户期待**：学生群体作为数字原生代，对线上化服务有强烈诉求

## 二、项目目标

### 2.1 愿景

构建高校学生事务"一站式"数字化管理平台，实现从"学生跑腿办事"到"数据跑路服务"的转变，让学生自主管理全过程线上闭环。

### 2.2 战略目标

| 目标 | 描述 | 衡量指标 |
|------|------|----------|
| 流程线上化 | 4大业务100%线上闭环 | 线上化率 ≥ 95% |
| 数据资产化 | 过程数据可追溯、可分析 | 电子档案生成率 100% |
| 管理智能化 | AI辅助决策与档案生成 | AI生成内容可用率 ≥ 80% |
| 体验一体化 | 学生"一站式"入口 | 学生满意度 ≥ 4.0/5.0 |

### 2.3 阶段性目标

| 阶段 | 时间 | 目标 | 交付标准 |
|------|------|------|----------|
| 短期（3个月） | W1-W12 | MVP上线，覆盖4大核心业务基础流程 | 3条核心用户旅程可走通 |
| 中期（6个月） | W13-W20 | 完善审批流、智能考勤、数据统计 | 通过安全审计+用户验收 |
| 长期（12个月） | W21-W40 | 接入AI能力，形成数字治理闭环 | AI评优推荐准确率 ≥ 75% |

### 2.4 业务KPI

| 业务域 | 核心指标 | 目标值 |
|--------|----------|--------|
| 团员发展 | 入团申请线上化率 | ≥ 95% |
| 团员发展 | 电子档案生成率 | 100% |
| 社团活动 | 活动审批平均周期 | ≤ 3个工作日 |
| 社团活动 | 活动签到准确率 | ≥ 98% |
| 社区自治 | 志愿时长自动统计率 | ≥ 90% |
| 勤工助学 | 工资核算自动化率 | ≥ 85% |
| 勤工助学 | 考勤异常率 | ≤ 2% |
| 全局 | 学生满意度 | ≥ 4.0/5.0 |

### 3.3 用户画像

**画像1：团委管理员 — 王老师**
- 年龄35岁，使用电脑办公为主
- 每学期需处理200+入团申请，审批链路长
- 期望：一键查看发展对象状态，自动生成电子档案
- 痛点：审批进度靠微信追问，积极分子状态靠Excel筛选

**画像2：普通学生 — 李同学**
- 大二学生，手机使用为主
- 同时参与社团活动和勤工助学
- 期望：一个入口完成所有事务，不用跑多个办公室
- 痛点：入团申请交纸质表、社团签到要排队、勤工考勤要刷卡

**画像3：社团负责人 — 张同学**
- 大三学生，管理50人社团
- 每月组织2-3次活动，审批流程繁琐
- 期望：活动策划在线提交、快速审批、自动统计参与率
- 痛点：活动策划要送纸质版给老师签字，参与人数靠手动数

### 6.2 团员发展模块

#### FR-L01 入团申请管理

| 属性 | 说明 |
|------|------|
| 功能描述 | 学生在线提交入团申请，团委管理员在线审批 |
| 输入 | 申请人基本信息、申请理由、个人简历 |
| 处理逻辑 | 1. 学生填写申请表 → 2. 提交至团委管理员待审批队列 → 3. 团委管理员审批（通过/驳回）→ 4. 通过后自动创建积极分子记录 |
| 输出 | 审批状态通知、积极分子登记 |
| 业务规则 | 1. 每位学生仅可提交一次入团申请<br>2. 驳回后可重新提交<br>3. 审批采用一级审批（MVP） |

#### FR-L02 积极分子管理

| 属性 | 说明 |
|------|------|
| 功能描述 | 管理积极分子信息，记录培养过程 |
| 输入 | 积极分子基本信息、培养记录（思想汇报、培训记录、实践经历） |
| 处理逻辑 | 1. 入团审批通过自动创建积极分子 → 2. 团委管理员可手动添加培养记录 → 3. 积极分子可自行提交思想汇报 |
| 输出 | 积极分子列表、培养记录时间线 |
| 业务规则 | 1. 积极分子状态：培养中/已发展/已淘汰<br>2. 培养记录不可删除，仅可追加<br>3. 思想汇报需团委管理员确认 |

## 八、MVP范围

### 8.1 MVP定义原则

1. **核心路径优先**：每个业务域选择1条最核心的端到端流程
2. **数据闭环**：确保核心流程产生的数据可存储、可查看
3. **最小可用**：跳过复杂审批流，采用简单的一级审批
4. **AI预留**：数据模型预留AI字段，但不实现AI功能

### 8.2 MVP功能范围

| 模块 | MVP包含 | MVP不包含 |
|------|---------|-----------|
| **团员发展** | 入团申请提交、一级审批、积极分子登记、培养记录增删改查 | 政审备案流程、电子档案自动生成、多级审批 |
| **社团活动** | 活动策划提交、一级审批、活动签到（二维码）、活动图片上传 | 活动总结归档、数据统计看板、AI活动总结 |
| **社区自治** | 自治队伍CRUD、纳新发布与报名、值班排班表 | 换届管理、志愿时长自动统计、个人成长档案 |
| **勤工助学** | 岗位发布、学生报名、录用审核、简易考勤 | 智能考勤（定位签到）、工资核算自动化、数据统计 |
| **基础能力** | 登录/注册、JWT认证、角色权限、文件上传（MinIO） | 数据导出、消息通知、操作日志 |

```
学生浏览岗位 → 报名 → 学工处审核录用 → 日常考勤打卡
```

### 10.4 风险与应对

| 风险 | 概率 | 影响 | 应对策略 |
