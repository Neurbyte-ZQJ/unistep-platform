# ADR-004：为什么采用 JWT 认证

> 状态：Accepted  
> 日期：2026-06-05  
> 决策范围：用户认证与会话机制

## 背景

本项目包含系统管理员、团委管理员、学工处管理员、社区管理员、社团指导老师、社团负责人、自治队伍负责人、普通学生等多个角色。系统需要在前后端分离架构下完成登录认证、角色识别、接口鉴权和菜单权限控制。

前端应用由 Vue3 构建，后端提供 Go REST API。用户登录后，前端需要在后续请求中携带认证信息，后端根据用户身份和角色权限判断是否允许访问相关接口。

## 决策

系统采用 **JWT（JSON Web Token）认证机制**。

- 用户登录成功后，后端签发 JWT。
- 前端保存令牌，并在后续请求中通过 `Authorization: Bearer <token>` 携带。
- 后端中间件解析 JWT，获取用户身份、角色或权限上下文。
- RBAC 权限校验在后端接口层强制执行。
- JWT 有效期按项目安全要求控制，必要时后续引入刷新令牌机制。

## 备选方案

### 方案一：Session + Cookie

Session + Cookie 是传统 Web 系统常用方案，服务端可控性强，便于主动失效会话。

但在前后端分离和 REST API 场景下，Session 需要服务端保存会话状态，后续多实例部署时还需要引入 Redis 等共享会话存储，增加架构复杂度。

### 方案二：OAuth2 / OpenID Connect

OAuth2 / OIDC 适合统一身份认证、第三方登录和多系统单点登录场景。

但本项目 MVP 阶段主要是自有账号体系，暂未接入统一身份认证平台。直接引入 OAuth2 / OIDC 成本较高，流程复杂度超过当前需求。

### 方案三：API Key

API Key 适合服务间调用或开放平台接口。

但本项目是面向个人用户的业务系统，需要表达用户身份、角色和权限，API Key 不适合作为主要用户认证机制。

## 优点

1. **适合前后端分离**：JWT 可由前端在 API 请求中携带，不依赖服务端 Session。
2. **无状态认证**：后端无需保存会话，便于未来水平扩展 API 服务。
3. **实现简单**：Go 生态中 JWT 中间件和库成熟，易于集成 Gin。
4. **权限上下文清晰**：Token 中可包含用户 ID、角色标识等基础声明，便于后端快速构建鉴权上下文。
5. **跨端友好**：浏览器、移动端 Web、后续小程序或 App 都可使用统一 Bearer Token 机制。

## 缺点

1. **主动失效较复杂**：JWT 一旦签发，在过期前默认有效；如需立即踢下线或禁用，需要引入黑名单或版本号机制。
2. **令牌泄露风险**：前端存储 Token 需要防范 XSS、日志泄露和不安全传输。
3. **权限变更实时性有限**：如果角色权限写入 Token，权限变化可能要等 Token 过期后才完全生效。
4. **Token 体积问题**：不应在 JWT 中放入过多权限和业务数据，否则会增加请求开销。

## 结论

本项目采用 **JWT 认证**。JWT 与前后端分离架构、REST API 和后续多端访问模式匹配，能够以较低复杂度满足 MVP 阶段的认证需求。系统应避免在 Token 中存放敏感信息，并在后端强制执行权限校验。若后续需要单点登录、统一身份认证或更强会话控制，可再接入 OAuth2/OIDC 或引入刷新令牌与 Token 黑名单机制。
