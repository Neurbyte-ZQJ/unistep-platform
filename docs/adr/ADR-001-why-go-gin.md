# ADR-001：为什么选择 Go + Gin

> 状态：Accepted  
> 日期：2026-06-05  
> 决策范围：后端语言与 Web 框架选型

## 背景

本项目是面向高校学生事务的“一站式”自主管理过程管理系统，覆盖团员发展、社团活动、社区自治、勤工助学等核心业务。系统需要支持统一认证、RBAC 权限、审批流转、文件上传、统计查询等能力，并要求 MVP 阶段快速交付、部署简单、运行稳定。

高校信息化场景通常存在预算有限、运维团队规模较小、并发峰值集中在报名、审批、签到等时间段的特点。因此后端技术栈需要兼顾开发效率、性能、部署复杂度和后续演进空间。

## 决策

后端采用 **Go + Gin** 作为主要开发技术栈。

- Go 作为后端开发语言。
- Gin 作为 HTTP Web 框架。
- REST API 作为前后端通信方式。
- 后续结合 GORM、JWT、SQLite、MinIO 等组件完成业务能力建设。

## 备选方案

### 方案一：Java + Spring Boot

Java + Spring Boot 生态成熟，适合大型企业系统，具备完善的权限、安全、事务和工程化能力。

但对于本项目 MVP 阶段而言，Spring Boot 项目启动成本、运行资源占用和工程复杂度相对更高，对小团队快速交付不够轻量。

### 方案二：Node.js + Express/NestJS

Node.js 适合快速构建 API，前后端都使用 JavaScript/TypeScript 时协作成本较低。NestJS 在工程结构和依赖注入方面较完善。

但 Node.js 在 CPU 密集型任务和长期后台服务稳定性方面需要更多工程约束；高校后台管理系统对部署稳定性和单二进制交付更敏感。

### 方案三：Python + FastAPI/Django

Python 生态丰富，FastAPI 开发效率高，Django 管理后台能力强。

但 Python 运行时部署、性能调优和并发处理通常需要额外依赖 ASGI/Gunicorn/Uvicorn 等组件，整体部署链路不如 Go 简洁。

## 优点

1. **部署简单**：Go 可编译为单个二进制文件，便于在高校内网服务器或 Docker 环境中部署。
2. **性能充足**：Go 原生并发模型适合处理活动报名、二维码签到、审批查询等突发请求。
3. **框架轻量**：Gin API 简洁，路由、中间件、参数绑定、JSON 响应等能力满足 REST API 开发需要。
4. **工程复杂度可控**：相比重型框架，Go + Gin 更适合 MVP 快速落地，同时保留清晰分层架构。
5. **生态成熟**：Go 生态中有 GORM、JWT、日志、配置、对象存储 SDK 等常用组件，可支撑项目主要需求。
6. **后续演进友好**：如未来拆分服务或接入更高并发能力，Go 具备较好的扩展基础。

## 缺点

1. **脚手架能力较弱**：Gin 本身较轻量，不像 Spring Boot 提供大量开箱即用能力，需要团队自行约定项目结构。
2. **泛型与业务抽象成本**：Go 的表达能力相对 Java/TypeScript 更克制，复杂业务抽象需要谨慎设计。
3. **后台管理生态不如 Java 完整**：权限、工作流、报表等企业级组件需要更多自研或组合实现。
4. **团队学习成本**：如果团队主要经验来自 JavaScript、Java 或 Python，需要适应 Go 的错误处理和工程习惯。

## 结论

本项目后端选择 **Go + Gin**。该组合在性能、部署、开发复杂度和后续扩展之间取得了较好平衡，适合本项目“快速交付 MVP、低成本运维、后续可演进”的目标。对于当前高校学生事务管理系统的规模和业务复杂度，Go + Gin 能够提供足够稳定、轻量且可维护的后端基础。
