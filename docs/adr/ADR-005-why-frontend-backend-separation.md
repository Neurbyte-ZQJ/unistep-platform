# ADR-005：为什么采用前后端分离

> 状态：Accepted  
> 日期：2026-06-05  
> 决策范围：系统总体架构模式

## 背景

本项目需要建设高校学生事务“一站式”管理平台，覆盖 PC 管理端和学生侧 Web 使用场景。系统包含多角色权限、审批流程、表单提交、二维码签到、文件上传、统计查询等能力。

项目需求文档要求系统采用 Vue3 前端应用与 Go REST API 服务。前端关注页面交互、路由、权限菜单和用户体验，后端关注认证鉴权、业务规则、数据访问、文件存储和安全控制。

## 决策

系统采用 **前后端分离架构**。

- 前端使用 Vue3 构建单页应用，负责页面展示、交互、路由和状态管理。
- 后端使用 Go + Gin 提供 REST API，负责认证、权限、业务逻辑和数据访问。
- 前后端通过 HTTP JSON API 通信。
- 用户认证采用 JWT Bearer Token。
- 前后端可独立开发、测试、构建和部署。

## 备选方案

### 方案一：传统服务端渲染单体架构

由后端直接渲染 HTML 页面，例如使用模板引擎输出页面。

该方案部署简单，首屏加载快，适合页面交互较少的系统。但本项目包含大量动态表单、角色菜单、审批状态、二维码签到和管理端交互，服务端渲染会使前后端职责耦合，降低迭代效率。

### 方案二：全栈同构框架

例如 Nuxt、Next.js 等框架可同时处理前端页面和服务端能力。

该方案适合内容型站点、SEO 场景或统一全栈团队。但本项目核心是内部业务管理系统，SEO 不是重点，后端需要承载明确的业务服务和权限控制，独立 API 更符合系统边界。

### 方案三：低代码/无代码平台

低代码平台可快速搭建表单、审批和管理页面。

但本项目涉及多业务域、权限模型、后续 AI 能力预留、文件存储和定制化流程，低代码平台可能在复杂规则、代码可控性和长期维护方面受限。

## 优点

1. **职责清晰**：前端专注用户体验和页面交互，后端专注业务规则、数据和安全。
2. **开发并行**：前后端可基于 API 协议并行开发，提高 MVP 交付效率。
3. **多端复用**：同一套后端 API 后续可服务移动端 Web、小程序、App 或第三方系统。
4. **技术演进独立**：前端和后端可分别升级框架、组件库和工程体系。
5. **适合权限系统**：后端统一进行接口鉴权，前端根据权限控制菜单和按钮展示。
6. **部署灵活**：前端静态资源与后端 API 可独立部署，也可通过 Nginx 或网关统一转发。

## 缺点

1. **接口协作成本**：需要维护 API 规范、字段约定、错误码和联调流程。
2. **认证安全复杂度增加**：需要处理 Token 存储、跨域、过期刷新和接口鉴权。
3. **首屏性能需优化**：单页应用可能需要路由懒加载、资源压缩和缓存策略。
4. **部署组件更多**：相比传统单体，前端静态资源、后端 API、反向代理等部署链路更多。

## 结论

本项目采用 **前后端分离架构**。该架构与项目的多角色管理、复杂表单交互、REST API 后端服务和未来多端复用需求匹配。虽然会带来接口协作和认证安全方面的额外工作，但对于“一站式”学生事务平台的长期演进、团队分工和用户体验建设而言，前后端分离是更合适的架构选择。
