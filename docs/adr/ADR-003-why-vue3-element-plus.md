# ADR-003：为什么选择 Vue3 + Element Plus

> 状态：Accepted  
> 日期：2026-06-05  
> 决策范围：前端框架与 UI 组件库选型

## 背景

本项目面向高校学生事务管理，前端需要支持多角色、多模块、多表单、多审批页面和数据列表。系统既要服务团委、学工处、社区管理员等 PC 端高频办公用户，也要兼顾学生在移动端 Web 上提交申请、报名、签到和查询状态。

MVP 阶段需要快速构建登录、角色菜单、申请表单、审批列表、活动管理、岗位管理、签到等中后台能力。因此前端技术栈需要具备成熟生态、组件丰富、开发效率高、团队学习成本可控等特点。

## 决策

前端采用 **Vue3 + Element Plus**。

- Vue3 作为主前端框架。
- Vite 作为构建工具。
- Element Plus 作为 UI 组件库。
- Pinia 作为状态管理方案。
- 前端通过 REST API 与 Go 后端通信。

## 备选方案

### 方案一：React + Ant Design

React + Ant Design 生态成熟，中后台组件能力强，社区资源丰富。

但 React 对团队工程约定和状态管理方案选择要求更高，组件写法相对灵活，若团队 Vue 经验更充分，采用 React 可能增加学习和协作成本。

### 方案二：Vue2 + Element UI

Vue2 + Element UI 生态成熟，历史项目多，学习成本低。

但 Vue2 已不再是新项目首选，长期维护和生态演进不如 Vue3。Element UI 也不适合面向未来的新系统建设。

### 方案三：原生 HTML/CSS/JavaScript 或轻量框架

原生方案依赖少，适合非常简单页面。

但本项目包含大量表单、表格、弹窗、权限菜单和交互状态，使用原生方案会显著增加开发和维护成本。

## 优点

1. **适合中后台系统**：Element Plus 提供表单、表格、分页、弹窗、菜单、上传、日期选择等常用组件。
2. **与 Vue3 深度适配**：Element Plus 面向 Vue3 设计，生态兼容性好。
3. **开发效率高**：可快速搭建管理后台页面，满足 MVP 快速交付目标。
4. **学习成本可控**：Vue 模板语法直观，适合中小团队协作。
5. **工程生态成熟**：Vue3 + Vite + Pinia 组合已成为主流方案，文档和社区资源充足。
6. **用户体验一致**：Element Plus 提供统一视觉和交互规范，减少自定义组件成本。

## 缺点

1. **视觉同质化**：Element Plus 默认风格容易产生常见后台系统外观，需要适当主题定制。
2. **移动端体验有限**：Element Plus 主要面向 PC 中后台，学生移动端页面需要额外适配或补充移动端组件。
3. **复杂交互需二次封装**：审批流、动态表单、权限菜单等场景仍需项目内封装。
4. **组件库依赖较强**：长期使用后，页面结构和交互可能较深绑定 Element Plus。

## 结论

本项目选择 **Vue3 + Element Plus** 作为前端基础技术栈。该组合非常适合以表单、表格、审批、管理后台为主的高校学生事务系统，能够在保证开发效率的同时降低团队协作成本。对于学生移动端高频场景，应在 Vue3 基础上进行响应式适配，必要时补充更适合移动端的组件方案。
