# ADR-002：为什么选择 SQLite

> 状态：Accepted  
> 日期：2026-06-05  
> 决策范围：MVP 阶段主数据库选型

## 背景

本项目需要存储用户、角色权限、入团申请、社团活动、活动报名、签到记录、自治队伍、勤工助学岗位、考勤记录等结构化数据。MVP 阶段重点是验证核心业务闭环，预期部署环境以高校内部服务器或单机 Docker Compose 为主。

项目文档中对用户规模的预估为全校级别，约不超过 5 万用户，4 年数据量预计不超过 10GB。系统主要负载集中在查询、审批、报名、签到等典型业务操作，初期不需要复杂的分布式数据库能力。

## 决策

MVP 阶段采用 **SQLite** 作为主数据库。

- 使用 SQLite 存储核心结构化业务数据。
- 通过 GORM 进行模型映射与数据访问。
- 数据库以单文件形式部署和备份。
- 后续如并发规模、数据量或多节点部署需求增长，可迁移至 PostgreSQL。

## 备选方案

### 方案一：PostgreSQL

PostgreSQL 功能强大，支持复杂查询、事务、并发控制、索引、JSON、全文检索等能力，适合长期演进。

但 MVP 阶段引入 PostgreSQL 会增加数据库安装、运维、备份、权限管理和连接池配置成本。对于初期单体部署和有限数据规模而言，能力较充足但复杂度偏高。

### 方案二：MySQL

MySQL 生态成熟，运维人员熟悉度高，适合常见 Web 系统。

但 MySQL 同样需要独立数据库服务和运维配置。相比 SQLite，MVP 阶段的部署门槛更高，且本项目没有必须依赖 MySQL 特性的需求。

### 方案三：MongoDB

MongoDB 适合文档型数据和灵活字段模型。

但本项目核心数据具有明显关系结构，如用户-角色、活动-报名-签到、岗位-报名-考勤等，更适合关系型数据库。使用 MongoDB 会增加关系查询和事务建模复杂度。

## 优点

1. **零配置部署**：SQLite 不需要单独安装数据库服务，降低本项目 MVP 部署难度。
2. **单文件易备份**：数据库文件可直接备份、迁移，适合高校内部小规模运维场景。
3. **满足数据规模**：预估用户量和数据量在 SQLite 可承载范围内。
4. **开发效率高**：本地开发、测试和演示环境无需额外数据库依赖。
5. **成本低**：减少数据库服务器、DBA 运维和环境配置成本。
6. **迁移路径清晰**：通过 GORM 抽象数据访问层，后续可迁移至 PostgreSQL。

## 缺点

1. **写并发能力有限**：SQLite 适合轻中等并发场景，高写入并发下可能出现锁竞争。
2. **不适合多节点共享写入**：如果后续部署为多实例 API 服务，SQLite 单文件模式会成为架构瓶颈。
3. **高级数据库能力有限**：相比 PostgreSQL，复杂权限、分析查询、并发控制和扩展能力较弱。
4. **运维监控生态较弱**：SQLite 缺少独立数据库服务常见的监控、审计和在线运维能力。

## 结论

本项目 MVP 阶段选择 **SQLite**。该选择符合“快速启动、低成本部署、满足初期规模”的目标。考虑到项目当前重点是验证学生事务线上闭环，而不是构建高并发分布式数据平台，SQLite 是合理的初期数据库方案。若后续出现多校区部署、高并发写入、复杂统计分析或多节点扩展需求，应优先迁移至 PostgreSQL。
