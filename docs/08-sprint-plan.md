## 1. Sprint规划总览

本计划以“先基础、再核心流程、后数据智能”为原则，将项目拆分为 8 个 Sprint。每个 Sprint 默认按 2 周规划，工时以人时（h）估算，用于项目排期、任务分配和验收管理。

| Sprint | 主题 | 核心交付 | 预估工时 |
|--------|------|----------|----------|
| Sprint1 | 项目初始化 | 前后端工程骨架、数据库初始化、部署基础、UI框架 | 160h |
| Sprint2 | 用户权限模块 | 登录注册、JWT认证、RBAC权限、用户角色管理 | 180h |
| Sprint3 | 团员发展模块 | 入团申请、审批、积极分子、培养记录 | 220h |
| Sprint4 | 社团活动模块 | 活动创建审批、报名、二维码签到、活动图片上传 | 240h |
| Sprint5 | 自治队伍模块 | 队伍管理、纳新报名、报名审核、值班排班 | 220h |
| Sprint6 | 勤工助学模块 | 岗位发布、岗位报名、录用审核、考勤打卡 | 230h |
| Sprint7 | 统计分析模块 | Dashboard、业务统计、异常统计、数据报表 | 200h |
| Sprint8 | AI能力模块 | AI活动总结、AI培养档案、AI评优推荐、成长档案 | 220h |
| **合计** | - | - | **1670h** |

## 2. Sprint1：项目初始化

### 目标

完成项目基础工程、技术框架、开发规范和部署基础，为后续业务模块开发提供稳定底座。

### 任务

| 编号 | 任务 | 说明 | 工时 |
|------|------|------|------|
| S1-T01 | 后端工程初始化 | 创建 Go + Gin 项目结构，规划 handler/service/repository/model 分层 | 20h |
| S1-T02 | 前端工程初始化 | 创建 Vue3 + Vite + Element Plus + Pinia + Router 项目结构 | 20h |
| S1-T03 | 数据库初始化 | 配置 SQLite、GORM、AutoMigrate 或初始化脚本 | 24h |
| S1-T04 | 基础配置管理 | 环境变量、配置文件、日志基础配置 | 16h |
| S1-T05 | 统一响应与错误码 | 实现统一 JSON 响应、错误码、分页结构 | 16h |
| S1-T06 | 前端基础布局 | 实现登录前后布局、侧边导航、顶部栏、基础路由 | 24h |
| S1-T07 | 文件目录与规范 | 建立前后端目录规范、命名规范、接口调用规范 | 12h |
| S1-T08 | Docker Compose 基础 | 准备前端、后端、MinIO/本地存储的部署骨架 | 20h |
| S1-T09 | 健康检查接口 | 实现 `/health` 接口和基础连通性验证 | 8h |
| **合计** | - | - | **160h** |

### 验收标准

1. 前端项目可启动，并展示基础登录页和主布局。
2. 后端项目可启动，`/health` 接口返回正常。
3. SQLite 数据库可正常连接，基础表迁移无错误。
4. 前后端可通过 `/api/v1` 完成基础联通。
5. Docker Compose 可启动基础服务。
6. 项目目录与架构文档中的分层设计一致。

### 风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 前后端目录规范不统一 | 后续模块开发混乱 | Sprint1 明确目录、命名和分层约束 |
| SQLite 初始化约束不完整 | 后续数据一致性问题 | 初始化阶段开启外键、WAL、索引和检查约束 |
| MinIO 部署复杂度超预期 | 阻塞文件上传能力 | 保留本地文件存储降级方案 |
| 基础架构过度设计 | 拖慢业务迭代 | MVP 采用模块化单体，避免提前微服务化 |

## 3. Sprint2：用户权限模块

### 目标

完成统一认证、JWT鉴权、RBAC角色权限和动态菜单能力，保障后续业务模块具备安全访问控制基础。

### 任务

| 编号 | 任务 | 说明 | 工时 |
|------|------|------|------|
| S2-T01 | 用户模型实现 | users、roles、permissions、user_roles、role_permissions | 24h |
| S2-T02 | 登录注册接口 | 账号密码登录、注册/账号创建、密码哈希 | 28h |
| S2-T03 | JWT认证 | JWT签发、解析、过期校验、认证中间件 | 24h |
| S2-T04 | RBAC鉴权 | 权限编码、角色权限校验、接口级权限控制 | 32h |
| S2-T05 | 用户管理页面 | 用户列表、创建、编辑、启停用 | 24h |
| S2-T06 | 角色权限页面 | 角色列表、权限分配、角色绑定用户 | 24h |
| S2-T07 | 前端权限路由 | 登录态保持、动态路由、动态菜单、按钮权限 | 28h |
| S2-T08 | 登录安全策略 | 登录失败计数、账号锁定、退出登录 | 16h |
| **合计** | - | - | **180h** |

### 验收标准

1. 用户可通过学号/工号和密码登录系统。
2. 登录成功后返回 JWT、用户信息、角色和权限列表。
3. 未登录访问受保护接口返回 401。
4. 无权限访问接口返回 403。
5. 不同角色登录后展示不同菜单和操作按钮。
6. 系统管理员可维护用户、角色和权限。
7. 密码使用 bcrypt 哈希存储，不保存明文密码。

### 风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 权限粒度设计过粗 | 后续业务越权风险 | 按“资源:动作”定义权限编码 |
| 只依赖前端菜单控制 | 存在接口越权 | 后端强制执行 RBAC 和数据权限校验 |
| JWT 主动失效能力弱 | 用户禁用后令牌仍可能有效 | 缩短有效期，关键接口实时校验用户状态 |
| 多角色权限冲突 | 菜单和按钮展示异常 | 用户多角色权限取并集，冲突规则前置定义 |

## 4. Sprint3：团员发展模块

### 目标

完成入团申请、团委审批、积极分子登记和培养记录闭环，实现团员发展核心流程线上化。

### 任务

| 编号 | 任务 | 说明 | 工时 |
|------|------|------|------|
| S3-T01 | 入团申请模型与接口 | league_applications 表、申请创建/提交/查询 | 30h |
| S3-T02 | 入团申请页面 | 学生申请表单、草稿、提交、进度查看 | 32h |
| S3-T03 | 入团审批接口 | 审批通过、驳回、审批意见、状态流转 | 28h |
| S3-T04 | 审批管理页面 | 待审批列表、审批详情、筛选分页 | 30h |
| S3-T05 | 积极分子自动登记 | 审批通过后创建 activists 记录 | 24h |
| S3-T06 | 积极分子管理 | 积极分子列表、详情、状态查看 | 24h |
| S3-T07 | 培养记录管理 | cultivation_records 新增、确认、时间线展示 | 36h |
| S3-T08 | 附件上传集成 | 思想汇报/证明材料附件上传与关联 | 16h |
| **合计** | - | - | **220h** |

### 验收标准

1. 学生可填写并提交入团申请。
2. 学生提交后可查看申请状态和审批意见。
3. 团委管理员可查看待审批列表，并执行通过/驳回操作。
4. 入团申请审批通过后自动生成积极分子记录。
5. 团委管理员可查看积极分子列表和详情。
6. 可为积极分子追加培养记录，并按时间线展示。
7. 入团申请唯一性、驳回后重提、培养记录仅追加等业务规则生效。

### 风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 团员发展流程后续变复杂 | 一级审批难以扩展 | Service 层封装状态机，预留多级审批扩展点 |
| 培养记录数据不可追溯 | 影响电子档案生成 | 培养记录仅追加，保留创建人和时间 |
| 附件材料类型不可控 | 文件安全风险 | 使用统一文件上传白名单与大小限制 |
| 申请重复提交 | 数据混乱 | 数据库唯一约束 + 业务层校验 |

## 5. Sprint4：社团活动模块

### 目标

完成社团活动从创建、审批、报名、二维码签到到活动图片上传的核心闭环。

### 任务

| 编号 | 任务 | 说明 | 工时 |
|------|------|------|------|
| S4-T01 | 活动模型与接口 | club_activities 创建、编辑、查询、状态流转 | 32h |
| S4-T02 | 活动创建页面 | 社团负责人创建活动策划、保存草稿、提交审批 | 28h |
| S4-T03 | 活动审批功能 | 指导老师审批通过/驳回、审批意见 | 28h |
| S4-T04 | 活动列表与详情 | 学生端活动浏览、详情查看、筛选 | 28h |
| S4-T05 | 活动报名功能 | 报名、取消报名、人数上限、重复报名校验 | 32h |
| S4-T06 | 二维码签到 | 签到码生成、签名、有效期、扫码签到 | 44h |
| S4-T07 | 签到统计 | 签到人数、签到列表、未签到列表 | 20h |
| S4-T08 | 活动图片上传 | 活动结束后图片上传、预览、关联活动 | 16h |
| S4-T09 | 移动端适配 | 活动列表、报名、扫码签到移动端体验优化 | 12h |
| **合计** | - | - | **240h** |

### 验收标准

1. 社团负责人可创建活动策划并提交审批。
2. 指导老师可审批活动，通过后活动进入报名中状态。
3. 学生可浏览活动、查看详情并报名。
4. 活动报名人数不得超过上限，每人每活动只能报名一次。
5. 社团负责人可生成签到二维码。
6. 学生扫码后可完成签到，且每人每活动只能签到一次。
7. 签到二维码具备有效期和签名校验。
8. 活动结束后可上传活动图片并正常预览。

### 风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 二维码被转发代签 | 签到数据失真 | 二维码短时效、签名校验、后续扩展位置校验 |
| 活动报名高峰并发写入 | SQLite 写锁冲突 | 控制事务范围，关键操作加唯一约束和重试机制 |
| 移动端扫码体验差 | 学生签到失败率高 | 移动端优先优化签到页面和反馈状态 |
| 活动状态流转复杂 | 出现非法操作 | 后端统一状态机校验，不依赖前端按钮控制 |

| S5-T02 | 队伍管理页面 | 队伍创建、编辑、停用、负责人设置、成员查看 | 32h |
| S5-T03 | 纳新公告接口 | recruitment_notices 发布、关闭、查询 | 24h |
| S5-T04 | 纳新发布页面 | 自治队伍负责人发布纳新公告 | 20h |
| S5-T05 | 纳新报名功能 | 学生查看纳新列表、提交报名 | 24h |
| S5-T06 | 报名审核功能 | 负责人/管理员审核通过或驳回 | 28h |
| S5-T07 | 值班排班模型与接口 | duty_schedules、排班创建、冲突检测 | 36h |
| S5-T08 | 值班排班页面 | 周/月排班表、人员选择、冲突提示 | 26h |
| **合计** | - | - | **220h** |

### 验收标准

1. 社区管理员可创建、编辑和停用自治队伍。
2. 队伍负责人必须为队伍成员。
3. 自治队伍负责人可发布纳新公告。

5. 负责人或管理员可审核纳新报名。
6. 审核通过后学生自动加入对应自治队伍。
7. 管理员可创建值班排班表，并对同一人员同一时间段冲突进行提示或阻止。

### 风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 队伍负责人和成员关系混乱 | 数据权限难以控制 | 建立 community_members 关系表并强制校验负责人归属 |
| 排班冲突规则不清晰 | 值班表不可用 | MVP 先实现同一人员同一时间段唯一约束 |
| 纳新名额超限 | 队伍人数失控 | 审核通过时校验 quota 上限 |
| 不同社区管理规则差异 | 需求频繁调整 | 将区域、队伍状态、纳新状态设计为可配置枚举 |

## 7. Sprint6：勤工助学模块

### 目标

完成勤工岗位发布、岗位报名、录用审核和考勤打卡能力，支撑勤工助学业务线上闭环。

### 任务

| 编号 | 任务 | 说明 | 工时 |
|------|------|------|------|
| S6-T01 | 岗位模型与接口 | work_positions 发布、编辑、下架、查询 | 30h |
| S6-T02 | 岗位管理页面 | 学工处管理员发布/编辑/下架岗位 | 28h |
| S6-T03 | 岗位列表与详情 | 学生端岗位浏览、筛选、详情查看 | 24h |
| S6-T04 | 岗位报名功能 | work_applications 报名、取消、状态查询 | 28h |
| S6-T05 | 录用审核功能 | 管理员录用/拒绝，生成 work_assignments | 34h |
| S6-T06 | 在岗唯一性校验 | 同一学生仅允许一个 active 在岗岗位 | 18h |
| S6-T07 | 考勤打卡接口 | 上班打卡、下班打卡、工时计算、异常标记 | 36h |
| S6-T08 | 学生考勤页面 | 今日打卡、本月工时、异常记录 | 24h |
| S6-T09 | 考勤汇总页面 | 管理员按月/岗位/学生查看考勤 | 28h |
| **合计** | - | - | **230h** |

### 验收标准

1. 学工处管理员可发布、编辑、下架勤工岗位。
2. 学生可浏览岗位列表、查看详情并报名。
3. 管理员可审核报名并录用学生。
4. 录用后系统生成在岗关系。
5. 同一学生同一时间只能存在一个在岗岗位。
6. 在岗学生可完成上班、下班打卡。
7. 系统可自动计算工时并标记异常。
8. 管理员可查看考勤汇总和异常记录。

### 风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 工资核算规则复杂 | 超出本 Sprint 范围 | Sprint6 只完成考勤和工时基础，工资核算放入统计/后续增强 |
| 考勤补录规则不统一 | 数据可信度下降 | MVP 仅允许管理员调整并记录调整人和备注 |
| 学生多岗位冲突 | 录用数据异常 | 应用层校验 + 数据库部分唯一索引 |
| 打卡时间规则差异 | 异常标记不准确 | 先实现基础迟到/早退/缺卡标记，复杂规则后续配置化 |

## 8. Sprint7：统计分析模块

### 目标

完成管理端 Dashboard、核心业务指标统计、异常数据分析和基础报表能力，为管理决策提供数据支撑。

### 任务

| 编号 | 任务 | 说明 | 工时 |
|------|------|------|------|
| S7-T01 | 指标口径定义 | 入团线上化率、审批周期、签到准确率、考勤异常率等 | 20h |
| S7-T02 | 统计接口设计 | 按模块、时间、院系、状态聚合查询 | 32h |
| S7-T03 | 管理端 Dashboard | 待办、关键指标、业务趋势、异常概览 | 36h |
| S7-T04 | 团员发展统计 | 申请量、审批通过率、积极分子数量、培养记录数量 | 24h |
| S7-T05 | 社团活动统计 | 活动数量、报名人数、签到率、活动审批周期 | 24h |
| S7-T06 | 自治队伍统计 | 队伍数量、纳新报名、值班出勤、志愿时长 | 24h |
| S7-T07 | 勤工助学统计 | 岗位数量、报名录用率、工时、异常考勤 | 24h |
| S7-T08 | 数据导出基础 | 关键列表 Excel/CSV 导出能力 | 20h |
| S7-T09 | 性能优化 | 统计查询分页、索引检查、慢查询优化 | 16h |
| **合计** | - | - | **200h** |

### 验收标准

1. 管理端 Dashboard 可展示待审批、异常、核心指标和趋势数据。
2. 团员发展、社团活动、自治队伍、勤工助学均有基础统计页面或卡片。
3. 指标计算口径与 BRD/PRD 中 KPI 定义一致。
4. 支持按时间、院系/部门、状态等维度筛选统计数据。
5. 核心统计接口响应时间满足 95% 请求 ≤ 500ms 的目标。
6. 关键数据列表支持基础导出。

### 风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 指标口径不统一 | 管理层不信任数据 | Sprint 开始前确认指标定义和计算公式 |
| 统计查询影响业务接口 | 高峰期系统变慢 | 统计接口分页、加索引，避免长事务 |
| 历史数据不足 | 趋势图价值有限 | MVP 先展示当前周期数据，后续随使用积累 |
| 数据导出涉及敏感信息 | 隐私合规风险 | 导出接口强制鉴权，必要字段脱敏 |

## 9. Sprint8：AI能力模块

### 目标

基于已沉淀的过程性数据接入 AI 能力，实现活动总结、团员培养档案、评优推荐和个人成长档案等智能辅助能力。

### 任务

### 验收标准

1. 系统可基于活动数据生成活动总结草稿。
2. 系统可基于培养记录生成团员培养档案草稿。
3. 系统可基于学生过程性数据生成评优推荐理由。
4. AI 生成内容必须经过管理员确认后才能归档或对外使用。
5. 学生个人成长档案可展示团员发展、社团活动、自治服务、勤工助学记录。
6. AI 生成内容可用率达到 ≥ 80%。
7. AI 调用过程具备日志记录，便于排查和成本统计。

### 风险

| 风险 | 影响 | 应对 |
|------|------|------|
| AI生成内容不准确 | 影响业务公信力 | AI 仅生成草稿，必须人工确认后使用 |
| 外部模型成本不可控 | 运维成本上升 | 记录调用次数和场景，限制高频调用 |
| 数据隐私泄露 | 合规风险 | 提示词脱敏，避免传输不必要敏感字段 |
| 过程性数据不足 | AI结果质量低 | 仅对数据完整度达标的对象开放生成 |
| 用户过度依赖AI | 业务判断弱化 | 页面明确标注“AI辅助生成，仅供参考” |

## 11. 跨Sprint风险总览

| 风险 | 影响范围 | 等级 | 应对策略 |
|------|----------|------|----------|
| 审批流程频繁变化 | 团员发展、社团活动、勤工助学、自治队伍 | 高 | 业务状态流转集中在 Service 层，预留审批流扩展 |
| SQLite 并发写入瓶颈 | 报名、签到、考勤 | 高 | 启用 WAL，控制事务范围，预留 PostgreSQL 迁移方案 |
| 移动端体验不足 | 学生端报名、签到、打卡 | 中 | 核心学生流程优先移动端适配 |
| 文件上传安全 | 所有附件和图片上传 | 中 | 类型白名单、大小限制、随机对象Key、受控访问 |
| 数据权限遗漏 | 全系统 | 高 | 后端强制 RBAC + 数据范围校验，不信任前端 |
| AI结果不可控 | AI总结、档案、推荐 | 中 | AI生成内容默认草稿，必须人工确认 |
| 业务数据口径不一致 | 统计分析 | 中 | Sprint7 前统一指标定义和计算公式 |

## 12. 验收管理建议

1. 每个 Sprint 结束必须完成演示验收，演示对象包括产品负责人、业务代表和技术负责人。
2. 每个业务模块至少覆盖“创建/提交、审批/审核、查询、状态流转、权限拦截”五类测试场景。
3. 所有接口必须满足统一响应格式和鉴权要求。
4. 所有列表页面必须支持分页，避免一次性加载大数据集。
5. 所有关键业务操作必须记录操作人、操作时间和状态变化。
6. 涉及学生个人数据的页面和接口必须执行数据权限校验。
7. AI能力上线前必须完成内容准确性、隐私合规和人工确认流程验收。
