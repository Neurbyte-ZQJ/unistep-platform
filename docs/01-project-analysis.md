## 一、项目分析

### 1.1 项目概述

本项目为高校团委、学生工作处、学生社区建设一套数字化管理平台，实现学生事务全过程线上化管理。系统覆盖团员发展、社团活动、社区自治、勤工助学四大核心业务领域，旨在替代传统纸质/Excel管理方式，构建"一站式"学生自主管理闭环。

### 1.2 现状痛点

| 痛点 | 具体表现 | 影响范围 |
|------|----------|----------|
| 流程碎片化 | 入团申请走纸质、审批走微信、档案存Excel，数据孤岛严重 | 团委、学生工作处 |
| 考勤不精准 | 社团活动签到靠手写、勤工助学考勤靠打卡机，易代签漏签 | 社团指导中心、勤工助学办 |
| 统计滞后 | 月报/季报靠人工汇总，数据延迟1-2周 | 各管理部门 |
| 档案缺失 | 学生毕业时缺乏完整过程性记录，评优推荐缺乏数据支撑 | 学生个人、辅导员 |
| 协作低效 | 多部门业务交叉，信息传递依赖线下会议和群消息 | 全校师生 |

### 1.3 项目目标

- **短期（3个月）**：完成MVP，覆盖4大核心业务的基础流程线上化
- **中期（6个月）**：完善审批流、智能考勤、数据统计，达到可交付状态
- **长期（12个月）**：接入AI能力，实现智能档案生成、评优推荐，形成数字治理闭环

### 1.4 技术选型合理性评估

| 技术 | 选型 | 合理性 |
|------|------|--------|
| 前端框架 | Vue3 + Vite | 生态成熟、开发效率高、适合中小团队 |
| UI组件库 | Element Plus | 企业级中后台组件完善，与Vue3深度适配 |
| 状态管理 | Pinia | Vue3官方推荐，TypeScript友好 |
| 后端语言 | Go | 高并发、编译型、部署简单、适合微服务演进 |
| Web框架 | Gin | 轻量高性能，Go生态主流选择 |
| ORM | GORM | Go生态最成熟的ORM，自动迁移便利 |
| 数据库 | SQLite3 | 零配置、单文件部署，适合高校场景（用户量≤5万） |
| 对象存储 | MinIO | S3兼容、自部署、适合图片/文档存储 |
| 认证 | JWT | 无状态、前后端分离友好 |
| 部署 | Docker Compose | 一键编排、环境一致、运维简单 |

> **SQLite3容量评估**：单数据库支持TB级存储，5万用户量+4年数据量预估≤10GB，完全满足。若后期需横向扩展可平滑迁移至PostgreSQL。

## 四、MVP范围界定

### 4.1 MVP定义原则

- **核心路径优先**：每个业务域选择1条最核心的端到端流程
- **数据闭环**：确保核心流程产生的数据可存储、可查看
- **最小可用**：跳过复杂审批流，采用简单的一级审批
- **AI预留**：数据模型预留AI字段，但不实现AI功能

### 4.2 MVP功能范围

| 模块 | MVP包含 | MVP不包含 |
|------|---------|-----------|
| **团员发展** | 入团申请提交、一级审批、积极分子登记、培养记录增删改查 | 政审备案流程、电子档案自动生成、多级审批 |
| **社团活动** | 活动策划提交、一级审批、活动签到（二维码）、活动图片上传 | 活动总结归档、数据统计看板、AI活动总结 |
| **社区自治** | 自治队伍CRUD、纳新发布与报名、值班排班表 | 换届管理、志愿时长自动统计、个人成长档案 |
| **勤工助学** | 岗位发布、学生报名、录用审核、简易考勤 | 智能考勤（定位签到）、工资核算自动化、数据统计 |
| **基础能力** | 登录/注册、JWT认证、角色权限、文件上传（MinIO） | 数据导出、消息通知、操作日志 |

### 4.3 MVP用户旅程（核心路径）

**旅程1：学生入团申请**
```
登录 → 填写入团申请表 → 提交 → 团委老师审批 → 成为积极分子
→ 记录培养过程 → （后续迭代：发展对象 → 政审 → 电子档案）
```

**旅程2：社团活动全流程**
```
社团负责人创建活动 → 指导老师审批 → 活动发布 → 学生报名
→ 现场扫码签到 → 上传活动图片 → （后续迭代：总结归档）
```

**旅程3：勤工助学申请**
```
学生浏览岗位 → 报名 → 学工处审核录用 → 日常考勤打卡
→ （后续迭代：工资核算 → 数据统计）
```

采用 **RICE评分法**（Reach × Impact × Confidence / Effort）：

- **Reach**：影响用户数（1-10）
- **Impact**：业务价值（1-5）
- **Confidence**：确定性（0.5/0.8/1.0）
- **Effort**：开发工作量（人周）

## 六、风险分析

### 6.1 风险登记册

| ID | 风险描述 | 概率 | 影响 | 等级 | 应对策略 |
|----|----------|------|------|------|----------|
| R1 | SQLite并发写入瓶颈（高校选课/报名高峰期） | 中 | 高 | 🔴高 | 写入队列+重试机制；预留PostgreSQL迁移方案 |
| R2 | MinIO部署复杂度超出预期（Docker网络问题） | 中 | 中 | 🟡中 | MVP阶段先用本地文件存储，迭代期切入MinIO |
| R3 | 审批流程需求频繁变更 | 高 | 中 | 🔴高 | 审批流引擎化设计，流程配置与业务逻辑解耦 |
| R4 | 学生端移动端体验差（Vue3非原生） | 中 | 中 | 🟡中 | 响应式布局优先；后续考虑UniApp封装 |
| R5 | 数据安全与隐私合规（学生个人信息） | 低 | 高 | 🟡中 | 敏感字段加密存储；操作审计日志；权限最小化 |
| R6 | 团委/学工处业务规则不统一（跨校区） | 中 | 低 | 🟢低 | 多租户数据隔离设计；业务规则可配置化 |
| R7 | AI功能依赖外部API，成本不可控 | 中 | 中 | 🟡中 | AI功能按需调用；支持多模型切换（本地/云端） |
| R8 | 二维码签到作弊（代签） | 高 | 中 | 🔴高 | 签到有效期+位置校验+人脸识别（P3迭代） |

     │            │            │
 低  │  R6        │            │
     └────────────┴────────────┴──→ 概率
